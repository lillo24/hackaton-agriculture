UI REPO CONFIG — Challenge 1 Hackathon

Goal
Build a demo-first frontend for Challenge 1.
Prioritize visual quality, clarity, and a convincing user flow over backend depth.
Use mock data unless a real integration is trivial and low-risk.

Core product idea
This is an agriculture alert platform UI.
The MVP should show:
- Farm Type selection
- Alerts Panel
- Alert Detail
- Config / Integrations

Main priority
We do NOT care about showing complex real code architecture.
We DO care about:
- cool graphics
- smooth animations
- strong visual hierarchy
- polished demo flow
- believable mock data
- easy deployment

Recommended stack
- React
- Vite
- TypeScript
- Tailwind CSS
- shadcn/ui
- Framer Motion
- React Router

Build principles
- Frontend only unless absolutely necessary
- Mock everything first
- Make the UI feel premium
- Keep components modular but do not overengineer
- Optimize for a hackathon demo, not production completeness

Design direction
Style:
- modern
- slightly futuristic
- agriculture + monitoring + operational feel
- clean but visually rich

Visual priorities:
- strong cards
- nice gradients
- soft shadows
- smooth hover states
- animated transitions
- good spacing
- readable typography
- clear severity colors for alerts

Use:
- subtle background textures or gradients
- animated counters / badges if useful
- animated page transitions
- micro-interactions on cards and buttons
- timeline / activity visuals in alert detail
- rich status indicators for integrations

Avoid:
- boring admin panel look
- too much plain table UI
- too much text density
- unnecessary forms
- backend-heavy features
- auth unless trivial

Data strategy
Use mock data for:
- farm types
- alerts
- alert severity
- alert sources
- integrations
- sync status
- recent history / timeline
- recommended actions

Mock data should feel realistic and agriculture-specific.

Farm types
At minimum:
- Vineyard
- Orchard
- Open Field
- Greenhouse

Rule:
Selected farm type should influence what alerts appear and what wording is shown.

Pages to build
1. Farm Type
- selection cards
- one active choice
- visually strong, almost like role/class selection
- this can be a page or section

2. Alerts Panel
- list of alerts
- filters or tabs if fast to add
- severity badges
- source badges
- compact but visually polished cards

3. Alert Detail
- reached by clicking an alert
- show:
  - summary
  - why triggered
  - source signals
  - recommended action
  - timeline/history
- make this page visually the richest

4. Config / Integrations
- show mock APIs and data sources
- card/list format
- status
- last sync
- available data types
- optional health indicators

Animation guidance
Use animation to improve perceived quality, not to distract.
Prefer:
- fade + slide on page enter
- card hover lift
- staggered list entrance
- animated badges or indicators
- subtle transitions between states

Do not:
- add long cinematic animations
- slow down navigation
- overuse bouncing effects

Architecture guidance
Keep it simple:
- reusable layout
- reusable cards/badges
- centralized mock data
- basic routing
- local state is enough unless needed

Suggested folders
- src/components
- src/pages
- src/data
- src/layout
- src/lib
- src/types

Demo mindset
Every screen should answer:
- what is happening
- why it matters
- what the user should do next

The demo should feel like:
"this product already knows the farm context and can guide action"

Definition of success
Success is NOT:
- real backend
- real API integration
- production-grade architecture

Success IS:
- polished UI
- clear navigation
- believable alert system
- coherent farm-type customization
- visually impressive prototype
- deployable in one day

Working rule for implementation
Whenever there is a tradeoff:
1. choose mock over real integration
2. choose polished visuals over deep infrastructure
3. choose demo clarity over technical completeness
4. choose fewer pages done well over more pages done badly
